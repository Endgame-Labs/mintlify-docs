<Warning>

  beta. Contact [support@endgame.io](mailto:support@endgame.io) to have the
  feature enabled for your organization.
</Warning>

The REST API uses **Bearer tokens** for authentication. Every request to
`/api/v1/*` must include a valid credential in the `Authorization` header.

```http
Authorization: Bearer eak_your_api_key_here
```

Two credential types are accepted:

- **User-scoped API keys** (prefix `eak_`) created from Endgame's API Keys
  settings page. These act as the user who created them.
- **Machine-to-machine (M2M) JWTs** issued by your organization's WorkOS
  OAuth client-credentials flow. These act as an organization-scoped service
  account.

## Create a user-scoped API key

<Note>
  You must be an admin in your Endgame organization to create API keys.
</Note>

<Steps>
  <Step title="Open the API Keys settings page">
    Sign in to Endgame and go to
    [https://app.endgame.io/settings/api-keys](https://app.endgame.io/settings/api-keys).
  </Step>
  <Step title="Create a new key">
    Click **Create API key**, give it a descriptive name (e.g.
    `integrations-backend-prod`), and save. The full key is shown **only
    once** -- copy it immediately and store it in a secrets manager.
  </Step>
  <Step title="Use the key in requests">
    Pass the key as a Bearer token on every API call:

    ```bash
    curl -X POST https://app.endgame.io/api/v1/threads \
      -H "Authorization: Bearer eak_your_api_key_here" \
      -H "Content-Type: application/json" \
      -d '{"prompt": "hello"}'
    ```
  </Step>
</Steps>

## Key scope and permissions

The two credential types have different permissions against the thread
resource:

| Capability                              | User-scoped key (`eak_`) | M2M JWT                       |
| --------------------------------------- | ------------------------ | ----------------------------- |
| Create a thread                         | ✅ Owned by the user     | ✅ Owned by an org-wide actor |
| Read / list the caller's own threads    | ✅                       | ✅                            |
| Read / list **other** users' threads    | ❌ `404 NOT_FOUND`       | ✅ All org threads            |
| Rename (`PATCH`) a thread               | ✅ Own threads only      | ❌ `403 FORBIDDEN`            |
| Delete a thread                         | ✅ Own threads only      | ❌ `403 FORBIDDEN`            |

M2M callers are designed for read-heavy integrations that should not mutate
user-owned content. If your integration needs to rename or delete threads,
use a user-scoped API key instead.

Cross-organization access is always denied -- a credential from one
organization cannot read or mutate threads in another.

## Obtain an M2M JWT

WorkOS M2M credentials are provisioned per-integration by the Endgame team.
If you're building a server-to-server integration, contact
[support@endgame.io](mailto:support@endgame.io) to request an M2M client and
exchange the resulting client credentials for a JWT via WorkOS's OAuth
token endpoint. Pass the returned `access_token` as the Bearer token on each
request.

## Revoke a key

If a user-scoped key is leaked or no longer needed, revoke it from the same
[API Keys settings page](https://app.endgame.io/settings/api-keys). Revocation
takes effect immediately; in-flight requests using the revoked key will fail
with `401 UNAUTHORIZED`.

To rotate M2M credentials, contact
[support@endgame.io](mailto:support@endgame.io).

## Security guidance

- Store keys in a secrets manager. Never commit them to source control.
- Use a separate key per environment (dev, staging, prod) so rotation is
  scoped.
- Give each key a descriptive name so revocation decisions are easy to audit.
- Rotate keys on a regular cadence and whenever a team member leaves.
