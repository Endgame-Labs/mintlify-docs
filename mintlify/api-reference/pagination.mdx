---
title: "Pagination"
description: "Cursor pagination for listing threads."
icon: "list"
---

`GET /api/v1/threads` is the only endpoint that paginates. It uses opaque
cursor pagination ordered by most recent activity first -- `updatedAt` when
present, falling back to `createdAt` -- with `id` as a tiebreaker. A thread
whose title was just updated jumps to the top, matching the order shown in
the Endgame app.

## Request parameters

| Parameter   | Type               | Default | Description                                                                        |
| ----------- | ------------------ | ------- | ---------------------------------------------------------------------------------- |
| `limit`     | `integer` (1-100)  | `25`    | Maximum number of threads to return on this page.                                  |
| `cursor`    | `string`           | --      | Opaque cursor from the previous response's `nextCursor`. Omit for the first page.  |
| `accountId` | `string`           | --      | Optional filter: only return threads associated with this account ID.              |

## Response shape

```json
{
  "threads": [
    { "id": "thr_...", "title": "...", "createdAt": "...", "...": "..." }
  ],
  "nextCursor": "<opaque-cursor-string>",
  "truncated": false
}
```

| Field        | Type               | Description                                                                                                                                                                                                              |
| ------------ | ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `threads`    | `Thread[]`         | Page of threads in most-recent-activity-first order.                                                                                                                                                                     |
| `nextCursor` | `string` or `null` | Cursor to pass on the next request. `null` when there are no more pages.                                                                                                                                                 |
| `truncated`  | `boolean`          | `true` when the organization has more threads than the server-side pagination window can walk. When this is `true`, narrow with `accountId` to reach threads the unfiltered walk can't surface. See [Truncation](#truncation). |

## Example

First page:

```bash
curl "https://app.endgame.io/api/v1/threads?limit=50" \
  -H "Authorization: Bearer eak_..."
```

Follow-up page using the cursor from the previous response:

```bash
curl "https://app.endgame.io/api/v1/threads?limit=50&cursor=<opaque-cursor-string>" \
  -H "Authorization: Bearer eak_..."
```

Continue until `nextCursor` is `null`.

## Truncation

The API walks the organization's most-recent threads in-memory up to a
fixed server-side window. When `truncated` is `true`, you've hit that
window and the cursor walk will stop before reaching every thread your
credential can see. Narrow the query with `accountId` to retrieve threads
the unfiltered walk can't surface. `truncated` is `false` for any org
whose total thread count fits inside the window (the common case during
beta).

## Invalid cursors

A malformed or corrupt cursor returns `400 INVALID_PARAMS` -- the API
will not silently fall back to the first page, because that would hide
data from the caller. Treat an invalid-cursor response as a programming
bug (e.g. the cursor wasn't URL-encoded), not a transient error, and
don't retry without fixing it.

## Tips

- Treat `cursor` and `nextCursor` as opaque. Their internal shape may
  change between releases; don't parse or construct them yourself.
- Pages are stable across concurrent thread creation -- a thread created
  mid-walk won't cause duplicates or skips for the remainder of the walk.
- URL-encode the cursor when passing it as a query parameter.
